## DECISION 01: Google Calendar Connection as "Optional but Blocking"

### 1. Context & Problem Statement
The technical requirements specify that users log in using Google and that the system must check their Google Calendar to prevent scheduling conflicts. However, it also states users "can connect their Google Calendar", implying it might not be strictly mandatory for initial application entry. We need to determine the exact boundary of when calendar access is enforced to balance security, user acquisition, and business rules.

### 2. Decision
We will decouple **Authentication** (Google Login) from **Authorization/Integration** (Google Calendar API Access). 
- **Authentication is mandatory:** Users must log in via Google to access the application (requesting only basic profile/email scopes).
- **Calendar Connection is "Optional but Blocking":** Users are not forced to grant Calendar permissions at the exact moment of signup. However, granting this permission is a strict prerequisite for executing the "Create Booking" command. 

### 3. Rationale
- **Security & Trust:** Requesting broad Calendar permissions upfront during initial login often leads to high drop-off rates. Requesting it only when the user is ready to extract value (booking) builds trust.
- **Domain Constraints:** Requirements **d(ii)**, **f**, and **h** mandate conflict validation against Google Calendar. Therefore, creating a booking without this connection violates core domain rules.
- **Separation of Concerns:** This approach cleanly separates the identity layer (Auth0/JWT via NestJS) from the external integration layer.

### 4. Consequences
- **Positive:** Reduces onboarding friction. Aligns with the principle of least privilege.
- **Negative/Complexity:** Requires the frontend to handle multiple user states (Authenticated vs. Authenticated + Connected) and the backend to explicitly validate calendar integration status on the booking endpoint.
