## DECISION 02: Soft Onboarding & Proactive UI Flow

### 1. Context & Problem Statement
Given Decision 01, a user can enter the application in a "Read-Only" state (logged in, but calendar not connected). Allowing the user to interact with booking forms only to be rejected by backend validation creates a frustrating "error-by-design" User Experience (UX).

### 2. Decision
We will implement a "Soft Onboarding" and proactive UI strategy comprising the following elements:
1. **Soft Onboarding:** Immediately after registration/login, prompt the user with an option to connect their calendar or "Skip for now".
2. **Read-Only State:** If skipped, the dashboard displays the schedule, but time slots are visually disabled (e.g., using `cursor-not-allowed` and reduced opacity).
3. **Visual Banner:** A persistent, clean banner is displayed prompting the user to connect their calendar to enable bookings.
4. **Action Interception (Upsell):** If a user clicks on a disabled slot, the system will not throw a standard error. Instead, it will trigger a friendly modal explaining the requirement and providing a direct "Connect Google Calendar" Call-to-Action (CTA).

### 3. Rationale
- **Preventive vs. Reactive:** Good UX communicates system state before the user commits an action. Visual blocking prevents wasted cognitive effort.
- **Product-Minded Engineering:** Handling the "click" with an upsell modal rather than a red error toast shifts the tone from penalizing the user to guiding them, demonstrating a strong understanding of product design alongside technical implementation.

### 4. Consequences
- **Positive:** High-quality, professional user experience that smoothly handles edge cases. 
- **Negative/Complexity:** Requires additional UI components (Banner, Upsell Modal) and state management in Next.js to toggle these views based on the user's connection status.
