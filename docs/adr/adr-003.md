## DECISION 03: Google Calendar Scope - Primary Calendar Only

### 1. Context & Problem Statement
Google Calendar allows users to maintain multiple calendars under a single account (e.g., personal, work, shared family calendars, holidays). The requirements state the system must "synchronize with Google Calendar to prevent booking over existing events," but do not specify if this applies to *all* calendars owned by the user or just the primary one. 

### 2. Decision
For the scope of conflict validation, the system will only synchronize with and check events against the user's **Primary Google Calendar**. Secondary, shared, or subscribed calendars will be ignored.

### 3. Rationale
- **API Efficiency:** The Google Calendar API provides a convenient `'primary'` alias for the main calendar. This allows the backend to perform a single, low-latency API call to fetch events, rather than querying the `calendarList` endpoint and performing complex aggregation or `freeBusy` queries across multiple IDs.
- **Scope Management:** Given the 72-hour time constraint, implementing multi-calendar selection UI and backend aggregation introduces unnecessary risk and scope creep.
- **User Expectation:** In most MVP scheduling applications, the primary calendar holds the most critical "blocking" events. 

### 4. Consequences
- **Positive:** Drastically simplifies backend integration (one standard REST call). Keeps the UI clean without needing a calendar selection settings page.
- **Negative:** If a user uses a secondary calendar for their main scheduling (which is uncommon but possible), the system might allow a booking conflict. This is deemed an acceptable trade-off for the MVP phase and can be noted as a future iteration point.
